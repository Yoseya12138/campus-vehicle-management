# 后端安全设计

## 1. 核心原则

- 二维码不保存学生或教职工个人明文信息。
- 数据库不保存二维码明文 token，只保存 `HMAC-SHA256(token)`。
- 所有敏感查询必须经过云函数，不允许前端直接查数据库。
- 权限判断放在云函数中，不能依赖小程序页面隐藏按钮。
- 手机号、学号、工号默认脱敏。
- 查看完整手机号、导出数据、管理人员变更、作废二维码必须写审计日志。

## 2. 角色与权限

| 角色 | 说明 | 权限 |
|---|---|---|
| user | 普通微信用户 | 提交车辆登记、查看本人车辆 |
| officer | 保卫科人员 | 扫码查询、登记违停、按权限查看完整手机号 |
| admin | 管理员 | 数据审核、人员管理、二维码作废/重置、数据导出 |

权限点：

```text
VIEW_FULL_PHONE   查看完整手机号
EXPORT_DATA       导出数据
MANAGE_OFFICER    管理保卫科/管理员账号
REVOKE_VEHICLE    作废或重置二维码
```

## 3. 二维码安全

二维码内容示例：

```text
EBIKE:v1:随机token
```

后端保存：

```text
tokenHash = HMAC-SHA256(token, TOKEN_PEPPER)
tokenSuffix = token 后 6 位
```

优点：

- 数据库泄露时无法直接还原二维码 token。
- 可通过更换 `TOKEN_PEPPER` 和批量重发码进行安全轮换。
- 后台列表只显示 `tokenSuffix`，不显示完整 token。

## 4. 接口限流

云函数对关键接口做限流：

- `createVehicle`：限制单用户每日登记次数。
- `resolveQr`：限制保卫科扫码频率。
- `exportData`：限制管理员导出频率。
- `manageOfficer`：限制人员管理频率。
- `bootstrapAdmin`：限制初始化尝试次数。

限流记录写入 `rateLimits` 集合，可设置 TTL 自动清理。

## 5. 审计日志

写入 `scanLogs` 的操作包括：

- 查看车辆详情；
- 查看完整手机号；
- 登记违停；
- 导出数据；
- 新增车辆。

写入 `securityEvents` 的操作包括：

- 初始化管理员；
- 新增/停用管理人员；
- 作废车辆二维码；
- 重置二维码 token；
- 接口异常；
- 触发限流。

## 6. 数据脱敏

默认返回：

```text
手机号：138****0000
学号：2023****456
工号：T202****001
```

只有具备 `VIEW_FULL_PHONE` 权限的保卫科人员或管理员才能查看完整手机号，且每次查看均记录日志。

## 7. 审核流程

学生或教职工自行提交车辆登记后：

```text
reviewStatus = pending
```

管理员审核后：

```text
approved / rejected
```

车辆被注销或二维码失效：

```text
status = revoked
```

## 8. 部署安全清单

- 设置强随机 `TOKEN_PEPPER`。
- 设置一次性 `BOOTSTRAP_CODE`，初始化后立即更换或删除。
- 数据库集合权限设置为前端不可直接读写。
- 管理人员使用真实 openid 授权。
- 定期导出审计日志留档。
- 定期复核 `officers` 权限人员。
- 数据导出必须要求填写导出原因并写日志。
- 车辆照片、违停照片不得公开访问。
- 离校、离职、车辆注销后及时停用数据。
